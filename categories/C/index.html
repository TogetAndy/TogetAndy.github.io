Introduction to Objects (background)

The genesis of the computer revolution was in a machine. The genesis of our programming languages thus tends to look like that machine.

This chapter introduce object-oriented programming (OOP)
include : overview of oop development methods.

1.1 The progress of abstraction

  • Abstracion

    • c /fortran /BASIC /primary abstraction still requires you to think in terms of the structure of the computer rather than the structure of the problem you are trying to solve.

    • Early languages such as LISP and APL chose particular views of the world (“All problems are ultimately lists” or “All problems are algorithmic”).

    • Each of these approaches is a good solution to the particular class of problem they’re designed to solve, but when you step outside of that domain they become awkward.

    • The object-oriented approach goes a step farther by providing tools for the programmer to represent elements in the problem space

  • The fisrt successful O-O language (the language c++ base on) Smalltalk’s characteristics :

    • Everything is an object.

    • A program is a bunch of objects telling each other what to do by sending messages.

    • Each object has its own memory made up of other objects

    • every object has a type

    • every object of particular type can receive the same mesages

1.2 An object has an interface

20:55

  1. although what we really do in object-oriented programming is create new data types, virtually all object-oriented programming languages use the “class” keyword. When you see the word “type” think “class” and vice vers
  2. a class describes a set of objects that have identical characteristics (data elements) and behaviors (functionality)
  3. The object-oriented approach is not limited to building simulations. Whether or not you agree that any program is a simulation of the system you’re designing, the use of OOP techniques can easily reduce a large set of problems to a simple solution.
  4. The interface establishes what requests you can make for a particular object. However, there must be code somewhere to satisfy that request. This, along with the hidden data, comprises the implementation.

1.3 The hidden implementation

  1. It is helpful to break up the playing field into class creators (those who create new data types) and client programmers
  2. the first reason for access control is to keep client programmers’ hands off portions they shouldn’t touch
  3. The second reason for access control is to allow the library designer to change the internal workings of the class without worrying about how it will affect the client programmer.
  4. the boundaries in a class: public, private, and protected.

reusing the implementation

  1. Code reuse is one of the greatest advantages that object-oriented programming languages provide.
  2. Once a class has been created and tested, it should (ideally) represent a useful unit of code.
  3. Composition is often referred to as a “has-a” relationship, as in “a car has an engine
  4. Inheritance, which is described next, does not have this flexibility since the compiler must place compile-time restrictions on classes created with inheritance.
